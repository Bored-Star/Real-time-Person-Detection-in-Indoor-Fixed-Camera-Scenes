# edge-video-system

这是一个端云协同架构的视频流人脸检测系统。系统将视频处理任务拆分到两台设备上：端侧设备负责视频采集、预处理和压缩发送，边缘侧设备负责接收视频流、执行人脸检测推理、目标跟踪和结果输出。这种架构适用于边缘计算场景，特别是需要将计算密集型的AI推理任务从资源受限的端侧设备卸载到性能更强的边缘服务器。

## 系统架构

系统的数据流向如下：视频文件输入到端侧设备A，经过视频解码和帧提取，然后进行预处理（调整尺寸、JPEG压缩）。压缩后的帧数据通过TCP连接发送到边缘侧设备B。设备B接收视频流，执行人脸检测推理，对检测结果进行多目标跟踪，最后将标注后的结果保存为图像文件。

设备A和设备B之间的通信采用自定义的二进制协议，每帧数据由4字节的大端序长度前缀和JPEG图像数据组成。接收端读取4字节获取数据长度，然后按照长度读取完整的JPEG数据。这种长度前缀的分帧方式避免了TCP流式传输中的数据粘包问题。

## 目录结构

```
edge-video-system/
├── device_a/                    # 端侧设备A模块
│   ├── capture.py              # 视频采集模块，负责从视频文件提取帧
│   ├── preprocess.py           # 预处理模块，调整图像尺寸和JPEG压缩
│   ├── sender.py               # 发送模块，处理TCP连接和数据传输
│   └── main.py                 # 设备A的主程序入口
├── device_b/                    # 边缘侧设备B模块
│   ├── receiver.py             # 接收模块，解析TCP数据流
│   ├── detector.py             # 人脸检测模块，加载模型执行推理
│   ├── tracker.py              # 目标跟踪模块，负责多目标跟踪和业务逻辑
│   ├── result_writer.py        # 结果输出模块，绘制检测框和统计信息
│   └── main.py                 # 设备B的主程序入口
├── test_videos/                # 测试视频文件目录
└── models/                     # 人脸检测模型文件目录
```

## 环境要求

项目需要Python 3.7或更高版本。主要依赖库包括opencv-python用于图像处理和视频编解码，numpy用于数值计算。设备B需要OpenCV的DNN模块支持，用于加载和运行深度学习模型。

人脸检测模型使用ResNet-SSD架构，需要两个文件：res10_300x300_ssd_iter_140000.caffemodel（模型权重）和deploy.prototxt（模型配置）。这些文件可以从OpenCV的官方模型仓库下载，或者使用项目中提供的脚本获取。

## 部署说明

这个系统设计为跨两台服务器运行，设备A和设备B可以部署在不同的物理机器上。在实际场景中，两台服务器可能都没有公网IP，这在GPU云服务器环境中很常见。这种情况下需要通过SSH端口转发建立通信隧道。

典型的部署场景是：设备A运行在有公网IP的跳板机上，设备B运行在内网的GPU服务器上。需要在跳板机上执行SSH端口转发命令，将本地端口的数据转发到GPU服务器的对应端口。

SSH隧道命令示例：

```bash
ssh -L 6006:localhost:6006 user@gpu_server_internal_ip
```

这个命令在跳板机上执行，会将跳板机的6006端口流量通过SSH隧道转发到GPU服务器的6006端口。设备A配置为连接localhost:6006，实际数据会通过隧道传输到GPU服务器上的设备B。

如果设备A也在内网环境中，可能需要建立双向的SSH隧道或者使用反向SSH转发。

## 运行方式

设备A（发送端）的启动命令示例：

```bash
python device_a/main.py --video test_videos/sample.mp4 --host localhost --port 6006 --fps 5 --size 640x480 --quality 80
```

常用参数说明：--video指定输入视频文件路径，--host和--port指定设备B的地址，--fps设置目标发送帧率，--size设置输出图像尺寸，--quality控制JPEG压缩质量（1-100）。

设备B（接收端）的启动命令示例：

```bash
python device_b/main.py --model-path models/res10_300x300_ssd_iter_140000.caffemodel --config-path models/deploy.prototxt --confidence 0.6 --dwell-threshold 10.0 --output output_frames
```

常用参数说明：--model-path指定模型权重文件，--config-path指定模型配置文件，--confidence设置检测置信度阈值，--dwell-threshold设置异常停留报警的时间阈值（秒），--output指定结果输出目录。

设备B需要先启动并进入监听状态，然后设备A再启动连接。设备A会按照指定的fps发送帧，设备B实时接收并处理。

## 已知问题与限制

目标跟踪模块在低帧率场景下存在一些限制。当设备A发送帧率较低（如5 FPS）时，同一目标在相邻帧之间的检测框位置跳变可能较大，导致IoU匹配失败。项目通过引入中心点距离辅助匹配策略缓解了这个问题，在IoU不满足主阈值的情况下，如果中心点距离较小且IoU满足辅助阈值，仍会认为是同一目标。但是这些阈值参数是针对特定测试视频调试得出的，换用不同分辨率或人员移动速度的视频时可能需要重新调整。tracker.py中的iou_threshold、iou_auxiliary_threshold和center_distance_threshold参数可以根据实际场景调节。

累计人流量统计目前依赖跟踪ID的分配，每当分配新的跟踪ID时累计计数加一。如果目标被遮挡、检测失败或位置跳变过大导致匹配失败，会被分配新的ID并重复计数。即使提高了max_unmatched_frames参数来增加对短暂遮挡的容错，仍无法完全消除这个问题。因此累计人流量统计数字是参考值而非精确值，主要适用于相对计数和趋势分析，不适合用于精确的人数统计场景。

异常停留报警功能已经通过构造的测试数据验证过，能够正确区分停留和移动场景并触发报警，但尚未在真实的长时间停留场景中验证。实际环境中的人体停留行为比测试场景更复杂，可能需要调整dwell_position的稳定判断阈值（当前设定为20像素）和停留时长的计算逻辑。

人脸检测模型本身使用的是2017年左右的ResNet-SSD架构，对侧脸、遮挡、极端光照等场景的检测效果有限。如果检测失败率较高，会影响整个系统的跟踪和统计准确性。可以考虑更换更新的检测模型来改善效果。

## 当前实现程度

设备A的视频采集、预处理、压缩和发送功能已经完整实现并测试过，能够稳定地按照指定帧率发送视频流。TCP传输协议的实现能够正确处理网络中断和重连，长度前缀的分帧方式在测试中没有出现数据粘包或丢失问题。

设备B的人脸检测模块使用了OpenCV DNN接口，能够正确加载Caffe格式的ResNet-SSD模型并执行推理。检测结果的输出格式标准化，包含边界框坐标和置信度。

目标跟踪模块实现了基于IoU的多目标跟踪算法，并且加入了中心点距离辅助匹配来处理低帧率场景。跟踪器能够为每个检测框分配唯一ID，跟踪目标的生命周期，并在目标消失一定帧数后清理过期跟踪。异常停留检测功能已实现基本逻辑，包括位置稳定性判断和停留时长计算。

结果输出模块能够将检测框、跟踪ID、人流量统计等信息绘制到图像上并保存为文件。输出的图像包含边界框、ID标签、累计人流量数字等可视化信息。

系统的端到端流程已经打通，能够在实际视频上运行并产生输出结果。但是在真实场景下的长期稳定性验证、参数调优、异常处理等方面还需要进一步完善。
